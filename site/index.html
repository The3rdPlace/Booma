This is Booma - a SDR VLF (and more) receiver

Booma was born out of curiosity while passing the SAQ / Grimeton transmitter en route to Oslo. The antennas got me wondering about what it was, and later how one would receive the tranmissions.

A few options existed for listening to SAQ Grimeton: SAQrx program by Johan, SM6LKM., and a (newer) fork of SAQrx SAQrx_V094, or a websdr receiver like this on in Twente.

Both SAQrx programs was windows programs, they actually worked with Wine on Linux, but I had the feeling that I wanted something a bit more recent, and futureproff - I also had some ancient experience with digital signalprocessing from my study days, and I had been looking for a project to get back into dsp programming.

So Booma was borne :)


Booma consist of 2 client applications and a server-side application for running a remote setup.
booma-console
Console (commandline) based receiver
booma-gui
GUI based receiver
booma-remote
Headless receiver frontend for a remote setup
Booma-console and booma-gui takes the same commandline parameters and use the same cached configuration, so they can be used interchangeably.

A quick start guide for listening to Grimeton with Booma: booma-[console|gui] -i AUDIO [input audio device] -f 17200 -o [output audio device]

These are recordings of the radio spectrum 0-24KHz with Grimeton transmitting. The recordings contains other VLF transmissions as well, so if you do not have an antenna and/or a soundcard ready for receiving yet, you can use these recordings for testing.
(check 16.4KHz, 22.1KHz and 23.4KHz - all rtty transmissions)

Listening to recording Booma:
Soundcard and AF output recordings
booma-[console|gui] -i [PCM|WAV] [filename] -f 17200 -o [output audio device] -it REAL -is AUDIO
Rtl-sdr recordings
booma-[console|gui] -i PCM [filename] -f 17200 -o [output audio device] -it IQ -is RTLSDR


Check which sounddevice to use for in- and output with booma-[console|gui] -c.

If you use linux and has PulseAudio running, you may need to fiddle a little bit with the configuration for the selected audio device so that PulseAudio do not block the device when you need to access it with Booma.
If you experiment too much and end up with some buggy setting, just flash you local stored settings using booma-[console-gui] -z and start over.


The digital signal processing is done by a lowlevel framework Hardt (The Hamradio DSP Toolkit), written as part of this project, but separated out since I want to be able to use it for other smaller projects as well (here).

Booma supports AM, SSB and CW receive modes. It can use audiocards or RTL-2832 dongles for input. Allthough Booma now supports the range from (almost) 0Hz to several GHz, the primary focus is as always - the Grimeton transmitter in Sweeden

You can run Booma entirely from a single pc, using the soundcard and some wire as an antenne - or you can scale it up a bit.

I have a Raspberry Pi (zero) in the basement with a USB soundcard and an rtl2832 dongle, both using a miniwhip antenna (original PA0RDT design and construction).

The Raspberry Pi is run from a good linear stabilized power supply, and wired directly to an ethernet switch - and thats important since a wifi ethernet card do generate quite a big burst of noise when transmitting, which ruins VLF reception.

In my shack elsewhere in the house, I run Booma on my pc (Ubuntu), receiving the antenna input from the Raspberry Pi. This way, I have moved the actual receiver as long away from in-house noise sources as possible (50Hz mains, the freezer, switched power adapters etc. etc.)


Booma requires the a few other software libraries. Most can be installed using apt, but the Hardt toolkit must - for now - be installed either from the prebuild deb(ian) package, or build and installed locally (which is not hardt.. got it!).
  • Hardt toolkit (The Hamradio DSP Toolkit) found here http://hardttoolkit.org/
  • PortAudio http://www.portaudio.com/
    • sudo apt install libportaudio2
    • sudo apt install libportaudio19-dev
  • Librtlsdr (controlling RTL-2832 dongles)
    • sudo apt install librtlsdr0
    • sudo apt install librtlsd-dev
  • Gcc, clang and cmake
    • sudo apt install gcc
    • sudo apt install clang
    • sudo apt install cmake
  • If you wants to build booma-gui also, then Fltk (graphical toolkat) and libjpeg
    • sudo apt install libfltk1.3
    • sudo apt install libfltk1.3-dev
    • sudo apt install libjpeg8
    • sudo apt install libjpeg8-dev
Hardt and Booma builds on Linux (tested on Debian and Ubuntu, but any distro should be fine), Mac and Raspberry Pi (with the Raspian OS, others may work too).

You can download a deb package for linux (debian or ubuntu) here: booma_{MAJORVERSION}.{MINORVERSION}-{BUILD}_amd64.deb and install with
  • sudo dpkg -i booma_{MAJORVERSION}.{MINORVERSION}-{BUILD}_amd64.deb
Build from the sources from the public Github repo at https://github.com/The3rdPlace/Booma.git
  • cmake .
  • make
  • sudo make install

Once you have installed Booma, you have three applications.
  • booma-console
  • booma-gui
  • booma-remote
All applications is essential the same receiver, but booma-remote does not have any command input once started, so if you operate a split setup, as I do, booma-remote is running on the Raspberry Pi, and booma-console is running on my pc in the shack.

To get started, check which sounddevices is available on your system and note which one is the input where the antenna is connected, and which one is the output:
  • booma-[console|gui|remote] -c
You can then start booma:
  • booma-[console|gui|remote] -i AUDIO (input device number) -o (output device number)
If you wish to run a remote setup, then it looks like this:
  • (on the remote system) booma-remote -i AUDIO (input device number) -s (dataport) (commandport)
  • (on the local system) booma-[console|gui] -i NETWORK (host) (dataport) (commandport)

If you want to use an RTL-2832 dongle, you can split the setup exactly as with a soundcard input, or run locally.

Check the RTL-2832 device number:
  • booma-[console|gui|remote] -r
then start booma:
  • booma-[console|gui|remote] -i RTLSDR (input rtl device number) -o (output audio device number)
If you run a remote setup with an RTL-2832 dongle, you must also tell the local booma what type of data and source you are using. This will typically by IQ for frequencies above 28MHz and Q for lower frequencies:
  • booma-[console|gui] -i NETWORK (host) (dataport) (commandport) -o (output audio device number) -it IQ|Q -is RTLSDR
There are other options as well, just run booma-[console|gui] -h to get help.

Important note: On some systems, PortAudio may not be accessible to regular users, so booma-[console|remote] -c will return 'no devices found', and trying to start the receiver will result in an error.
In that case, you either need to set booma-[console|remote] suid or run with sudo. The Raspian OS behaves like this, and other linux distros may have been configured this way too.
